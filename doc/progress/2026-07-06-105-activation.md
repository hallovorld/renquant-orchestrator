## Shadow acceptance bar (must be satisfied BEFORE treating this as deployment-ready)

None of this has been gathered yet as of this writing — the remaining-work list
below still starts at zero.

**A "clean session" is, concretely, a session-scheduler manifest where ALL of:**
1. `status == "completed"` (never `stopped_max_cycles`, `halted_kill_switch`,
   `halted_shadow_violation`, `halted_tick_error`, or `aborted_class_a_unavailable`)
2. `tick_count >= 1` (the loop actually ran at least one decision tick — a session
   that opened and closed with zero ticks proves nothing)
3. `tick_observer_errors` is absent or `0`
4. `kill_switch_engaged` is absent (the kill switch was never tripped)
5. `class_a` and `class_b` are both populated (the frozen signal loaded and the
   session-start snapshot was captured — a session that skipped straight to
   `PHASE_CLOSED` without ever reaching `PHASE_ENTRIES_OPEN` does not count)

**Required artifacts per clean session:** the session manifest itself
(`data/rq105/session_manifest_<date>.json` or wherever the scheduler's
`manifest_path` writes), the shadow tick log (`shadow.jsonl`), and the batch-scores
export + meta pair for that session's class-A signal.

**Success metric for the accumulation bar:** N consecutive clean sessions (N to be
set by the operator before the first paper-trading authorization — this doc does
NOT set N; that is a capital-risk-adjacent decision item #2 below, not an
engineering one).

**Rollback trigger:** any session that is NOT clean per the criteria above (a
`halted_*` or `aborted_*` status, a nonzero `tick_observer_errors`, or a kill-switch
trip) resets the clean-session streak to zero. A single non-clean session after N
clean ones does not "bank" progress — the streak requirement is consecutive.

## Round 1 (codex review)

STATUS: fixed
WHAT: three findings — (1) `DEFAULT_TICK_SECONDS` 720s->180s left the hardcoded
`max_cycles=60` in two `tests/test_entry_timing_policy.py` scheduler-loop tests
covering only ~3h of the fixture's 6.5h session, so they silently stopped
asserting `status == "completed"` and started passing on `stopped_max_cycles`
instead; (2) the fingerprint relaxation (`panel`, `global_calibration`,
`ranking.panel_scoring.artifact_path` required) had no schema distinguishing
"artifact not used by this config" from "artifact expected but missing", so a
future config referencing a new artifact whose hash silently failed could pass
undetected; (3) this doc's original "Infrastructure deployed" / "100% delivered"
framing overclaimed relative to the fact that zero shadow sessions had run.
WHY-DIR: codex correctly identified the tick-count assumption as a genuine
behavior break (not flaky), the fingerprint set as under-justified rather than
provably complete, and the doc as speaking with more authority than the
evidence earned.
EVIDENCE:
- (1) `tests/test_entry_timing_policy.py` now computes `max_cycles` from the
  fixture's actual session length divided by the scheduler's configured
  `tick_seconds` (plus a fixed safety margin), so it stays correct under any
  future cadence change instead of hardcoding a cadence-specific magic number.
  Confirmed both affected tests fail against the pre-fix hardcoded `max_cycles=60`
  and pass after.
- (2) Re-derived the required set to exactly `{panel, global_calibration}` — the
  two "primary runtime artifacts" `resolve_artifact_paths`
  (`RenQuant/backtesting/renquant_104/kernel/artifact_contract.py`) always
  aliases regardless of which underlying config-field variant produced them,
  which structurally proves they capture the actual panel-scoring model +
  calibrator used, i.e. the class-A signal's own inputs by this module's own
  definition. Dropped the redundant third key
  (`ranking.panel_scoring.artifact_path`), which duplicated `panel`'s value when
  present and spuriously required it on configs using the `panel_ltr.artifact_path`
  fallback instead (where that raw key is never even present). New tests prove:
  a bundle missing hashes for shadow-lane/ngboost/quality-floor/regime-conditional
  keys still passes; a bundle missing `panel` or `global_calibration` still fails.
  Applied identically to both `intraday_session_inputs.py` and
  `ops/renquant105/export_batch_scores.py` (duplicate logic, same bug, same fix).
- (3) Rewrote this doc's status/remaining sections above with a concrete,
  checkable acceptance bar instead of deployment-ready language.
