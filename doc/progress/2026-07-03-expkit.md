ROUND 2 (codex review): CI failure was a shallow-clone artifact, not a real freeze-first
          violation — `ci.yml`'s `test` job checkout had no `fetch-depth` (defaults to a
          depth-1 shallow clone), so `_first_commit_adding`'s `git log --diff-filter=A`
          could only ever see the single shallow tip commit for every file, making the
          spec and results look like they landed in the same commit. Reproduced locally
          (`git clone --depth 1` shows both files at commit 006dd2b; `git fetch
          --unshallow` then correctly shows spec at d860820, results at dbeb26b — genuinely
          different commits, spec strictly first). Fixed by adding `fetch-depth: 0` to the
          checkout step (matches what require-progress-doc.yml already does). Not a change
          to this PR's own code — a pre-existing CI-config gap this PR's new test was the
          first to expose.
          Scope concern (abstraction size for a substrate-constrained workflow): not
          re-scoping unilaterally. This is a consolidation of ALREADY-EXISTING duplicated
          logic (~8,800 script lines / 6+ re-implementations across the sprint), not new
          speculative capability — it explicitly does not touch production (BOUNDS: no
          production import, not a mass migration) and already caught one real bug during
          test-writing (the small-n exact-branch orientation bug, FOUND above). Flagging
          for codex to weigh in on whether that reasoning changes the scope call, rather
          than cutting library surface without full design context.

ROUND 3 (codex sharpened the scope ask; narrowed for real this time): codex named exactly
          two modules as clearly justified (prereg, evidence) and asked for "one or two
          [more] primitives that are already duplicated and already consumed," calling
          runner/verdict/plugins "ahead of demonstrated need." Investigated the actual claim
          per-module rather than trusting the round-2 summary uncritically:

          - prereg.py — KEPT. The frozen_spec.json write/read/hash convention is
            independently hand-rolled in scripts/d3_core_shrink_check.py (`frozen_spec.json`),
            scripts/m8_cluster_wave1.py + scripts/m8_independent_verification.py
            (`m8_frozen_spec.json`), scripts/s9_track_a_conditional.py. Note: the git-history
            freeze-first CHECK itself is genuinely new mechanization (the module's own
            docstring says so — "adds the missing mechanical check"), not consolidated
            duplicate code; kept anyway since it's the smallest possible addition that
            directly prevents the exact failure class this whole PR exists to catch.
          - evidence.py — KEPT, strong evidence. sha256_file/_json_default originate in
            scripts/c3_residual_momentum.py; scripts/msig_c2_quality.py does NOT
            re-implement them, it reaches back into c3 via a raw
            `importlib.util.spec_from_file_location` file-path load to reuse them —  a
            fragile ad-hoc cross-script import, arguably stronger evidence of real need
            than a clean duplicate would be.
          - evaluation.py — KEPT (codex's "one or two primitives" allowance; not explicitly
            named, but the evidence is as strong as evidence.py's). fwd_excess and
            per_date_ic are independently reimplemented in BOTH
            scripts/c3_residual_momentum.py and scripts/rs5_downcap_measurement.py;
            scripts/msig_c2_quality.py reaches into c3's copy via the same importlib.util
            hack as above.
          - stats.py — KEPT (the second of the "one or two primitives"). The core bootstrap
            (block_bootstrap_conditional_mean in c3_residual_momentum.py) is independently
            reimplemented THREE times under three different names:
            carried_mask_block_bootstrap (scripts/msig_c4_trendscan.py) and
            bootstrap_mask_removed_mean (scripts/rs5_downcap_measurement.py). The small-n
            exact-tail branch and multi-seed-unanimity logic in the same file are NOT
            duplication-justified in the same sense — they codify a method note (V3) and a
            named lesson (#264) rather than consolidate copy-pasted code. Kept anyway since
            splitting mid-file would fragment one coherent uncertainty-estimation module and
            the FOUND orientation bug (small-n exact-tail GO/KILL sign error) lives in this
            same file's exact-tail functions, making it directly load-bearing.
          - controls.py, verdict.py, runner.py, plugins/ — CUT. Real rationale exists for
            controls.py (S-REL R2, the λ-round-1 NULL-by-construction incident), but each
            experiment's actual plant/null CONSTRUCTION differs (RS-5: rank-z(label)+noise;
            D3: 0.9*rank(score)+0.1*rank(label); C4: blend-weight solved to a margin target;
            C2: PC-A z(label)+kappa*noise) — the shared element is a POLICY ("must plant AND
            null"), not duplicated CODE, so it doesn't clear the same bar as the four kept
            modules. controls.py also structurally imports GateDecision/Outcome from
            verdict.py, so cutting verdict.py required cutting controls.py too rather than
            leaving a half-working import. verdict.py's decide()/mde_one_sided() and
            runner.py's full pipeline are genuinely new decision machinery — no script
            reimplements a shared verdict FUNCTION, each experiment's verdict currently comes
            from bespoke prose reasoning over its own numbers. plugins/c2_quality.py only
            existed to demo runner.py, so it goes with it.
          Verified nothing outside expkit/ imports the four cut modules (a clean grep, zero
          hits) before removing them.
